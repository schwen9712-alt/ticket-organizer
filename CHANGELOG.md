## 2026-05-23 — 【紧急修复】关联订单逻辑过激进，把无关订单全串一起

**改了什么**
修复上一版引入的严重 bug：一次粘贴多个订单时，所有订单被错误地串成一个"关联组"，每个都标 🔗第N出 + 互相关联。

**bug 现象（用户截图）**
粘贴 6 个互不相关的订单（ZHANG/ZHOU、WANG、TAN/ZHAO、LIN、PENG/HU、HE），结果全被错误关联：
- 每单都标了 🔗第N出
- 每单的关联对象都是"ZHANG+WANG+TAN+LIN..."全部其他单
- 完全是乱的

**根因**
上一版的自动关联逻辑太蠢：
```
newOrders.forEach((o, i) => {
  o.ticketOrder = i + 1;          // 全部编号
  o.urgent = true;               // 全部加急
  o.linkedOrderIds = 所有其他单;  // 全部互相关联 ← 元凶
});
```
只要粘贴里有任何一单带"先开/再开"备注，就把整个批次全关联了。但实际上：多个不相关的订单经常一起粘贴，它们不该被关联。

**修复：保守关联规则**
两个订单只在【同时满足】以下条件时才关联：
1. 两个都带出票顺序备注（先开/再开...）
2. 共享至少一个航班（航班号+日期相同）或一个乘客

这样：
- ✓ PENG/HU（往返）和 HE（单程）共享 UA772 → 正确关联
- ✓ ZHANG/WANG/TAN/LIN 各自独立 → 不被关联
- ✓ 带💣但没有"先开"的单（如 WANG、TAN）→ 只标加急，不关联

用连通分组算法：从每个带 seqNote 的单出发，找所有与它共享航班/乘客的单，组成一组；只有组内 ≥2 单才建立关联。

**端到端测试（用户完整 6 单原始数据）全部通过**
- ✓ 6 单全部正确拆分
- ✓ 只有 #5(PENG/HU) ↔ #6(HE) 关联（共享 UA772）
- ✓ #1-#4 各自独立，无错误关联
- ✓ 💣 标记正确（#2/#3/#5/#6 有，#1/#4 无）

**改动位置**
- index.html 11215 附近：自动关联逻辑从"全批次关联"重写为"保守的共享航班/乘客判断"

**风险或注意事项**
- ⚠ 现在关联更保守：如果两个真正相关的单【既不共享航班也不共享乘客】，不会自动关联（但这种情况罕见，且可手动点🔗关联）
- ⚠ 不再强制整组 urgent：每单的💣加急按自己解析的来，不传染给同批次其他单
- ⚠ 配合上一版的"①先出/②后出"标签，关联组的顺序显示清晰无歧义

**回滚方式**
从 .backups/ 找上一版覆盖。⚠ 回滚会恢复"全批次乱关联"的 bug，强烈不建议。
