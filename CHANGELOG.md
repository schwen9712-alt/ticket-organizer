## 2026-05-23 — 已出票面板加「清除历史截图」按钮（释放 localStorage 空间）

**改了什么**
已出票（TICKETED）面板的 section-title 右侧，「＋ 手动补录」按钮旁边新增「🗑 清除历史截图」按钮。

点击后：
1. 扫描所有已出票订单的 `screenshots` 数组
2. 弹 confirm 显示：将清除多少张、多少个订单、估算释放多少空间（KB/MB）
3. 用户确认后，把所有 ticketed 订单的 `screenshots` 设为空数组
4. 写本地存储 + 推 Firebase 同步（其他设备也会同步清掉）
5. Toast 提示释放的空间

**为什么改**
用户反馈：顶部出现「⚠ 本地存储已满，数据仅保存在内存中」红色横幅。

根因：localStorage 浏览器硬上限通常 5-10MB。`settings` 字段里塞着 `ticketed`（823 单的完整数据，每单可能带 6 张 base64 JPEG 截图）。已出票订单的截图本质上是出票前给客户/代理看的临时凭证，归档后基本不需要再看，是最值得清理的部分。

**已出票截图清理是安全的**
- ✓ 订单本身（PNR、票号、乘客、价格、卡主、利润）全部保留
- ✓ 待出票订单的截图**完全不动**（出票前还要参考）
- ✓ 不影响 Excel 导出、报表生成、利润统计
- ✓ Firebase 端同步删除，所有设备一致

**改动位置**
- `index.html` 行 3431 附近：HTML 已出票面板顶部加按钮
- `index.html` 行 22253 附近：新增 3 个函数
  - `_estimateTicketedScreenshotsBytes()` — 估算待清理字节数（base64 → bytes × 0.75）
  - `_formatBytes(b)` — 字节数格式化（B / KB / MB）
  - `clearTicketedScreenshots()` — 主入口：估算 → confirm → 清空 → 存盘 → 同步 → 重渲染

**风险或注意事项**
- ⚠ **此操作不可撤销**：截图被删后就没了。但订单数据全在。confirm 弹窗里写清楚了
- ⚠ Firebase 同步是这次设计的一部分（清完会推云端，**其他设备的对应截图也会消失**）。如果你想"只清本机不清云端"，告诉我下一版改
- ⚠ 字节数是估算（base64 编码膨胀 ~33% 反推回原始字节，~3% 精度误差），仅用于给用户参考"释放了多少空间"，不影响实际清理效果
- ⚠ 治标不治本：localStorage 满的根因是 `settings.ticketed` 累积过大（823 单 + 全部业务字段）。等再涨到几千单还会满。**长期方案**是把 ticketed 从 settings 里拆出来单独存（之前讨论过的 A 方案）。如果以后还遇到同样问题，再做拆分

**回滚方式**
从 `.backups/` 找上一版 `index.html` 覆盖即可。新加的函数和按钮都是纯新增，没动现有逻辑，100% 安全回滚。
